# Правила таксономии: Бакеберг + новые таксоны

## Цель

- Сохранить **классическую систему Бакеберга** (роды и виды из его трудов).
- **Не терять** новые роды и виды, описанные после Бакеберга — вписывать их в дерево.
- **Ничего не терять из источников:** подвиды, разновидности, формы из выгрузок (GBIF и др.) должны попадать на сайт — в **одной карточке вида** показывать все его подвиды, разновидности и формы, чтобы была видна связь между близкородственными таксонами. Данные используем с цитированием источника (GBIF, UNAM) — это допускается условиями использования.
- В **синонимах** показывать **полную историю названий** (2–4 имени), а не только «старое → новое».
- **Не сливать** всё в одно: у коллекционера должны быть и «старички», и «новички» с пометкой происхождения.

---

## Цель карточки: всё про кактус в одном месте

В карточке вида должны быть по возможности: **название** (и принятое, и синонимы/история), **подвиды, разновидности, формы** (все инфравидовые таксоны — не объединять, не терять), **морфология** (стебель, колючки, цветок, плод), **фото** (если есть ссылки в источниках). Данные подтягиваем из разных источников (GBIF, cactus_cards, NCBI) постепенно; дубликаты обрабатываем осторожно.

**Правило про дубликаты и вариации:** Подвиды, разновидности, формы — это не дубликаты, а разные таксоны. Их **не объединять и не удалять**. Дубликатом считаем только **полное совпадение** одной и той же записи (одно и то же название в одном ранге). При слиянии источников вариации сохраняем: если у вида уже есть infraspecific из GBIF и пришли ещё данные из другого источника — дополняем, не заменяем.

---

## Почему сначала было «мало» видов

В выгрузке GBIF — **64 рода** и **638 видов**. В дереве Бакеберга изначально только **35 родов**. Скрипт `gbif_to_one_system.py` добавлял виды **только в те роды, которые уже были в taxonomy.json**. Поэтому **231 вид** из GBIF (роды Ariocarpus, Astrophytum, Echinocereus, Cochemiea и ещё 41 род) некуда было добавить — они «терялись».

**Что сделано:** скрипт `scripts/gbif_add_missing_genera.py` создаёт для всех родов из GBIF, которых нет в дереве, новую трибу **«Прочие роды (по GBIF)»** в подсемействе Cactoideae, добавляет в неё эти роды и создаёт для каждого файл `data/species/<род>.json` с видами из выгрузки. В итоге **ничего не теряется**: все 638 видов из GBIF отображаются на сайте (80 родов, 656 видов с учётом уже существовавших данных).

## Кто «старички», кто «новички»

**Источник данных:** `data/processed/species.csv` (NCBI + USDA), около **2500** записей видов/подвидов.

Два способа подсчёта:

1. **По полному имени** — старичок только если вид есть в нашем дереве (`data/species/*.json`, 18 видов) или его принятое имя/синоним в NCBI совпадает с одним из этих 18. Даёт **~0,8% старичков** (неестественно мало), потому что не учитывается смена рода: у Баккеберга был *Lobivia backebergii*, в NCBI — *Echinopsis backebergii*; это один и тот же вид.
2. **По эпитету (вторая часть названия)** — старичок, если эпитет вида встречается среди эпитетов из нашего дерева и из **всех** названий в `synonyms.csv` (там много старых родов: Lobivia, Echinocactus, Cereus, Trichocereus и т.д.). Если эпитет совпадает — вид считаем «старичком», просто сменившим «фамилию» (род). Даёт **~67% старичков, ~33% новичков**.

- **Старички (по эпитету)** — вид из NCBI, чей эпитет есть у Баккеберга/в синонимах (тот же таксон, возможно под другим родом).
- **Новички** — эпитет не встречается в этом наборе; это либо новые описания (post-Backeberg), либо названия, которых нет в нашей базе синонимов.

### Пропорция (по эпитету)

| Группа | Количество | Доля |
|--------|-------------|------|
| **Старички** (эпитет есть у Баккеберга/в синонимах) | **~1429** | **~67%** |
| **Новички** (эпитет не найден) | **~701** | **~33%** |
| **Всего записей в выборке NCBI** | **~2130** | 100% |

Пересчёт: `python3 scripts/report_old_new_species.py` (выводит оба способа).

### Пример одного «новичка»

**Rhipsalis rhombea** (в NCBI: род Rhipsalis, вид rhombea). В список старичков не попал, потому что эпитет **rhombea** не встречается ни в нашем дереве, ни среди названий в `synonyms.csv`. То есть мы не нашли у Баккеберга (и в текущей базе синонимов) вида с таким эпитетом — либо это более позднее описание, либо синоним под другим именем в нашу выгрузку не попал. Такие виды вписываем в дерево как «Новое открытие (post-Backeberg)» в ближайшую по смыслу группу (здесь — подсемейство Cactoideae, триба Rhipsalideae, род Rhipsalis).

### Вписывание «новичков» по подсемействам

Вписывать 701 «новичка» не кучей, а **по одному подсемейству**. Начать с самого простого, где меньше путаницы:

- **Pereskioideae** — два рода (Pereskia, Maihuenia), мало видов; удобно для старта. Дальше по очереди: Opuntioideae, затем Cactoideae (крупное, по трибам).

### Названия из книг Баккеберга (PDF)

Из тома 1 «Die Cactaceae» по **текстовому слою PDF** автоматически удалось вытащить только **несколько** названий (скрипт `scripts/extract_backeberg_names_from_pdf.py`): разметка и шрифты в PDF не позволяют надёжно выцепить все биномы со всех страниц. Чтобы получить полный список названий, которые упоминал Баккеберг, нужен **алфавитный указатель (Index)** из последнего тома или ручной/полуручной разбор. После появления такого списка его можно сравнить с NCBI и уточнить пропорцию старички/новички.

## Как вписывать новые таксоны в дерево

- Дерево остаётся в **`taxonomy.json`**: семейство → подсемейства → трибы → роды. Роды из Бакеберга **не удаляются** и **не объединяются** в один список с новыми без разметки.
- **Новые роды** (которых не было у Бакеберга):
  - добавляются в подходящую **трибу** (по современной классификации NCBI или по морфологии);
  - у узла ставится флаг **`"postBackeberg": true`** и при отображении подпись: **«Новое открытие (post-Backeberg)»**.
- **Новые виды**:
  - если род уже есть в дереве — вид добавляется в соответствующий `species/*.json` или в расширенный список с пометкой `postBackeberg`;
  - если род новый — сначала добавляется род (как выше), затем вид.
- Размещение «по смыслу»: по **роду из NCBI** (parent_id в `species.csv` / `genera.csv`) находим современный род, затем по таблице соответствия «NCBI genus ↔ наш узел дерева» помещаем в нужную трибу/род. Если род в нашем дереве есть — вид в этот род; если рода нет — создаём новый узел с `postBackeberg: true`.

## Синонимы: несколько названий (2–4 и больше)

- У одного таксона часто было **несколько названий** по мере переименований (например: *Cactus* → *Mammillaria* → подвиды).
- В данных и на карточке нужно показывать **цепочку имён**, а не только одно «ранее назывался».
- В **`synonyms_bridge.json`** для видов и родов:
  - поле **`nameHistory`** — массив строк в хронологическом порядке: `["старое название 1", "название 2", "текущее принятое"]`;
  - при наличии — оставить и **`synonyms`**, и **`previouslyCalled`** для обратной совместимости, но основной источник «истории» — **`nameHistory`**.
- На карточке: выводить все элементы `nameHistory` (или 2–4 имени), например: «Ранее: … ; … ; сейчас принято: …».

## Полнота выгрузки и что делать дальше

### Из этой выгрузки GBIF (UNAM) мы взяли всё

- **Виды** (ранг species) — все уникальные попали в `gbif_species.csv` и затем в дерево (в существующие роды или в новые через `gbif_add_missing_genera.py`).
- **Подвиды, разновидности, формы** — все записи с рангами subspecies, variety, form привязаны к родительскому виду и отображаются в карточке вида.
- Записи с рангом «genus» или «family» в occurrence — это не виды, а категории; их не нужно «вытаскивать» как отдельные таксоны.

**Итог:** из текущего архива GBIF мы собрали всё, что имеет смысл для карточек и дерева.

### Что можно сделать дальше (по желанию)

1. **Другие источники**  
   - Другие выгрузки GBIF (другие коллекции/фильтры по Cactaceae) — дадут дополнительные виды и проверку названий.  
   - У нас уже есть **`data/processed/species.csv`** (NCBI + USDA) — около **2500** видов; их можно постепенно сопоставлять с деревом и добавлять в нужные роды (по одному подсемейству).

2. **Проверка названий**  
   - Сверить ключевые названия с авторитетными источниками (Plants of the World Online, WCSP, или тот же GBIF Taxonomy) — чтобы исправить опечатки и устаревшие имена.

3. **Распределение по веткам**  
   - Сейчас 45 родов из GBIF стоят в одной трибе **«Прочие роды (по GBIF)»**. Когда решите, что список источников и проверка достаточны, их можно **распределить по правильным трибам** (Echinocereeae, Cacteae и т.д.) по современной классификации — тогда дерево станет и полным, и ботанически аккуратным.

**Рекомендация:** сначала при желании поискать другие выгрузки и/или подключить виды из `species.csv`, проверить названия, потом один раз распределить «прочие роды» по веткам — так не придётся перекладывать дважды.

## Файлы и скрипты

- **`data/taxonomy.json`** — дерево; при добавлении новых родов/триб — флаг `postBackeberg`.
- **`data/species/<genus>.json`** — виды по родам; при добавлении видов не из Бакеберга — у элемента вида можно добавить `"postBackeberg": true`.
- **`data/synonyms_bridge.json`** — мост имён; расширить форматом **`nameHistory`** (массив из 2–4+ названий).
- **`scripts/report_old_new_species.py`** — пересчёт и вывод пропорции старички/новички (по полному имени и по эпитету).
- **`scripts/extract_backeberg_names_from_pdf.py`** — извлечение биномиальных названий из PDF Баккеберга (Vol. 1); результат в `data/processed/backeberg_vol1_names.tsv`. Для полного списка нужен Index из последнего тома.
- **`scripts/build_synonyms_bridge.py`** — построение цепочек синонимов и запись **`nameHistory`**.

См. также: `data/README_CACTACEAE_SOURCES.md` — откуда берутся CSV и дерево.
