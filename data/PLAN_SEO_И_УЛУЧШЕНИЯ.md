# Оценка плана SEO и улучшений (по итогам обсуждения с Gemini)

## Как устроен сайт сейчас

- **Одна страница** классификации: `classification-cacti.html`. Карточки видов открываются на этой же странице (дерево + боковая панель), данные подгружаются из `data/species/*.json`.
- Отдельных HTML-страниц по каждому виду **нет** — поэтому «1000 страниц с уникальными meta» в текущей архитектуре означают не 1000 файлов, а либо генерацию статических страниц (большой рефакторинг), либо обновление title/description через JavaScript при открытии карточки (Google не всегда хорошо индексирует динамически подменённый контент).
- Синонимы показываются текстом (nameHistory, synonyms); ссылок на другие карточки внутри сайта нет.

## Оценка пунктов плана

### 1. Уникализация текста: перевод с английского на русский

**Плюсы:** Уникальный контент для поисковиков, польза для русскоязычных пользователей.  
**Реальность:** Поля `morphology_stem` и т.д. часто приходят с en.wikipedia.org на английском. Массовый перевод 800+ видов вручную трудоёмок; автоматический перевод (API) возможен, но нужна вычитка.  
**Рекомендация:** Делать постепенно: при добавлении/обновлении видов переводить описание на русский; для уже существующих — скрипт с переводом по API и последующей вычиткой ключевых видов.

### 2. Интеллектуальная перелинковка: синонимы → внутренние ссылки

**Идея Gemini верная:** синоним, который совпадает с принятым названием вида в нашей базе, должен вести на карточку этого вида на нашем же сайте (не на страницу благодарностей). Так и для пользователя полезно, и для SEO (внутренние связи, «глубина» сайта).

**Как сделать при текущей вёрстке:**  
Карточки не имеют своих URL. Нужно:
- ввести формат ссылки на карточку вида, например: `classification-cacti.html#g=astrophytum&s=astrophytum-ornatum` или через `?genus=astrophytum&species=astrophytum-ornatum`;
- при загрузке страницы читать параметры и, если заданы род и вид, автоматически раскрывать дерево и открывать нужную карточку;
- в блоке синонимов: для каждого имени из nameHistory/synonyms проверять, есть ли у нас вид с таким принятым названием (по латинскому имени); если да — рендерить это имя как ссылку на соответствующую карточку (тот же URL с нужными genus/species).

**Реалистично и сильно для SEO:** сделать перелинковку и «глубокие» ссылки на карточки — приоритет.

### 3. Структурирование данных GBIF/UNAM (таблицы)

**Плюсы:** Удобнее читать, можно вынести «Ареал», «Семейство», «Статус» в отдельные поля.  
**Реальность:** Сейчас в карточке в основном одно текстовое описание. Чтобы сделать таблицу, нужно либо хранить в JSON отдельные поля (ареал, семейство, статус), либо парсить их из текста. Данные из GBIF в текущей выгрузке не всегда содержат такие поля в структурированном виде.  
**Рекомендация:** При следующем обновлении данных из GBIF/NCBI закладывать структурированные поля; в карточке выводить их таблицей. Для уже загруженных данных — по мере возможности дополнять.

### 4. SEO: meta-теги на «каждую страницу»

**Ограничение:** Отдельных страниц по видам нет. Варианты:
- **A)** Генерация статических HTML-страниц для каждого вида (отдельный URL на вид) — максимум для SEO, но большая смена архитектуры.
- **B)** Одна страница классификации, но с поддержкой URL вида (см. п. 2). В `<head>` по умолчанию — общий title/description; при открытии карточки — обновлять `document.title` и тег `meta name="description"` через JS и при необходимости обновлять адрес через `history.pushState` (чтобы у карточки был свой URL). Поисковики постепенно учатся учитывать такие смены, но индексация может быть слабее, чем у статических страниц.
- **C)** Оставить один общий title/description для страницы классификации, но усилить их формулировками: «систематика кактусов», «синонимы», «описание вида» и т.п.

**Рекомендация:** Сначала сделать (B): ссылки на карточки + обновление title/description при открытии карточки. Если позже захочется «максимум SEO» — планировать (A) как отдельный этап.

### 5. Блок «Смотрите также» (3 случайных вида того же рода)

**Плюсы:** Навигация, дольше время на сайте, больше внутренних переходов — полезно и для пользователей, и для SEO.  
**Реальность:** Легко реализовать в текуней логике: при открытии карточки вида подгружаем список видов того же рода из того же JSON, выбираем 3 случайных (исключая текущий) и выводим под карточкой кнопки/ссылки; по клику открываем соответствующую карточку (с учётом будущих URL из п. 2).  
**Рекомендация:** Делать сразу после или вместе с введением ссылок на карточки.

## Предлагаемый порядок (без «копания» всей базы за раз)

1. **Ссылки на карточки и перелинковка синонимов**
   - Ввести URL вида: `classification-cacti.html?genus=...&species=...` (или через hash).
   - При загрузке страницы — открывать карточку по параметрам, если они есть.
   - В блоке «Ранее / синонимы» и «Синонимы»: для каждого имени проверять, есть ли у нас вид с таким названием; если да — выводить его как ссылку на эту карточку. Остальные синонимы оставить текстом.

2. **«Смотрите также»**
   - В конце карточки вида блок: 3 случайных вида из того же рода (кроме текущего); по клику — открыть их карточку и при желании подставить URL.

3. **SEO одной страницы**
   - Усилить title и meta description страницы классификации: добавить «систематика кактусов», «синонимы», «описание вида» и т.п.

4. **Перевод морфологии на русский**
   - Постепенно: скрипт + ручная вычитка или перевод при следующем обновлении данных.

5. **Структурированные поля и таблицы**
   - По мере появления полей (ареал, семейство, статус) в данных — выводить их таблицей в карточке.

## Ответ на вопрос: сравнение с «простым дублированием Википедии»

Текущий вариант уже лучше дублирования: есть своя структура (дерево, роды, виды), свои данные (GBIF, NCBI, синонимы), указание источников и лицензий. План Gemini усиливает это за счёт:
- **уникального контента** (русский текст),
- **внутренней перелинковки** (синонимы → карточки),
- **удобной навигации** (ссылки на карточки, «Смотрите также»),
- **понятных поисковику URL** на карточки (если реализовать п. 2 и 4 из плана).

Страница благодарностей при этом остаётся для юридической чистоты; для навигации и SEO важны именно внутренние ссылки на карточки видов и родов. Это и заложено в приоритеты выше.
